# 06. 日常運用と CI 連携

この章では、実装後に tester をどう運用し続けるかを説明します。

## 1. 日常開発フロー

新しいクラス/関数を追加したら、以下を必ず実施します。

1. 仕様を確認する
2. 先にテスト観点を列挙する
3. Python/C++ の該当テストを追加する
4. ローカルで実行する
5. Pull Request で CI 結果を確認する

---

## 2. テスト観点の最小セット

各クラス/関数に対して、最低でも次を用意します。

- 正常系
- 境界値
- 異常系

Array 例:

- 正常: `size=5` で代入/参照が一致
- 境界: `size=0`、`index=0` での挙動
- 異常: `index=-1`, `index=size` でエラー

---

## 3. ローカル実行のルール

### 3.1 変更前後での基本実行

```bash
make test-py
make test-cpp
make test
```

### 3.2 早く確認したい時

- Python だけ変更: `make test-py`
- C++ だけ変更: `make test-cpp`

ただし最終的には `make test` を必ず実行。

---

## 4. VS Code Testing View の使い分け

### 4.1 ケース単位で素早く回す

- 失敗ケースだけを実行して原因調査
- 修正後に同ケースを再実行

### 4.2 最終確認は全件

- ケース単位実行は高速だが漏れが出る
- 最後は全件実行で回帰確認する

---

## 5. CI との整合を保つコツ

- CI で使うコマンドはローカルと同じにする
- 長いコマンドを workflow に直書きしない
- `Makefile` に寄せる

このプロジェクトでは:

- C++: `make test-cpp`
- Python: `make test-py`
- 全体: `make test`

---

## 6. 失敗時の一次切り分け

1. Python か C++ かを切り分ける
2. 失敗ケース名を特定する
3. 入力データと期待値の差分を確認する
4. テストの誤りか、実装の誤りかを判断する
5. 修正後に再実行する

---

## 7. レビュー時の観点

PR レビューでは次を確認します。

- 変更したライブラリに対応するテストが増えているか
- 正常系だけで終わっていないか
- パラメータ化で重複コードを減らしているか
- エラー時に原因が分かるアサーションか

---

## 8. 運用アンチパターン

避けるべき例:

- 手元で一度もテストせず PR を出す
- ケースを 1 つだけ追加して満足する
- 失敗ログを読まずに assert を消して通す
- C++ と Python の仕様整合を確認しない

---

## 9. この章の完了条件

- 日常運用フローを説明できる
- CI とローカルの役割分担を説明できる
- 失敗時の切り分け手順を自分で実行できる
